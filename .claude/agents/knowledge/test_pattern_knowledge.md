# テストパターン・ナレッジベース

このファイルは、テストケース仕様書自動生成エージェントが参照する標準テストパターン集です。

---

## [パターンA] 標準フォームバリデーション

### 適用対象
- 会員登録フォーム
- お問い合わせフォーム
- プロフィール編集フォーム
- 各種申込フォーム

### 必須確認観点

#### 1. 入力項目の基本検証
- **必須入力チェック**
  - 全ての必須項目を空にして送信 → エラーメッセージ表示
  - エラーメッセージの表示位置と文言の確認
- **型チェック**
  - メールアドレス: 不正な形式（@なし、ドメインなし等）
  - 電話番号: 数字以外の入力
  - 郵便番号: 桁数不足・超過
- **文字数制限**
  - 最小文字数 - 1（エラー）
  - 最小文字数（OK）
  - 最大文字数（OK）
  - 最大文字数 + 1（エラー）

#### 2. サニタイジング・セキュリティ
- XSSパターン入力（`<script>alert('test')</script>`）
- SQLインジェクションパターン（`'; DROP TABLE users;--`）
- 特殊文字の扱い（絵文字、全角スペース等）

#### 3. UI/UX観点
- フォーカス順序の適切性（Tabキーでの移動）
- エラー時のフォーカス位置（最初のエラー項目へ）
- 送信ボタンの二重送信防止（連打しても1回のみ）
- 送信中のローディング表示

#### 4. 期待結果の標準パターン
- **正常系**: 確認画面または完了画面への遷移、入力値の保持
- **異常系**: エラーメッセージ表示、画面遷移なし、入力値の保持
- **境界値**: 最小/最大値で正常動作、超過時はエラー

## [パターンB] 検索・一覧画面

### 適用対象
- 商品検索画面
- ユーザー一覧画面
- 記事一覧画面
- 履歴表示画面

### 必須確認観点

#### 1. 検索結果の表示
- **0件ヒット時**
  - 「該当する結果がありません」などの適切なメッセージ表示
  - エラーではなく、空の状態を明示
- **1件ヒット時**
  - 正しく1件のデータが表示される
  - ページネーション非表示（または無効化）
- **多数ヒット時**
  - ページネーションの動作確認
  - 1ページあたりの表示件数が正しい

#### 2. ページネーション
- **基本動作**
  - 次ページ・前ページボタンの有効/無効状態
  - ページ番号の直接指定
  - 最初のページ・最後のページへのジャンプ
- **データの整合性**
  - ページ遷移後もソート順が保持される
  - フィルタ条件が保持される
  - 同じページに戻ったとき、同じデータが表示される

#### 3. ソート機能
- **各カラムでのソート**
  - 昇順・降順の切り替え
  - ソート後のデータの正確性
  - ページネーションとの組み合わせ
- **複数カラムソート**（該当する場合）
  - 第1ソートキー、第2ソートキーの適用順序

#### 4. ローディング状態
- **検索実行時**
  - スピナーまたはローディング表示
  - 検索ボタンの無効化（二重実行防止）
- **完了時**
  - スピナー消滅後、データが正しく描画されている
  - 検索ボタンが再度有効化される

#### 5. 期待結果の標準パターン
- **正常系**: データが正しく表示される、件数表示が一致する
- **異常系**: エラーメッセージ表示、前の状態を保持
- **境界値**: 0件、1件、最大表示件数、ページング境界

## [パターンC] 状態遷移・承認フロー

### 適用対象
- 申請→承認→完了フロー
- 下書き→公開フロー
- 注文→決済→発送フロー
- ステータス管理が必要な画面

### 必須確認観点

#### 1. ステータス変更の整合性
- **ステータス遷移の順守**
  - 許可された遷移のみ実行可能
  - 不正な遷移は防止される（例: 下書き→完了は不可）
- **ステータス表示**
  - 現在のステータスが明確に表示される
  - ステータスに応じたアイコンや色分け

#### 2. ボタン制御
- **ステータス別の表示制御**
  - 下書き: 「編集」「申請」ボタン表示
  - 申請中: 「取り下げ」ボタンのみ表示
  - 承認済み: ボタン非表示またはグレーアウト
- **権限による制御**
  - 一般ユーザー: 申請ボタンのみ
  - 管理者: 承認・却下ボタン表示

#### 3. データの永続性
- **ステータス変更後の確認**
  - リロード後もステータスが保持される
  - 他のユーザーから見ても同じステータス
- **履歴の記録**
  - ステータス変更履歴が記録される
  - 変更者、変更日時が記録される

#### 4. 並行操作の制御
- **排他制御**
  - 複数ユーザーが同時に同じデータを編集できないか
  - 編集中の表示（ロックアイコン等）
- **楽観的ロック**
  - 他ユーザーが先に更新した場合の警告

#### 5. 期待結果の標準パターン
- **正常系**: ステータス遷移成功、適切なメッセージ表示、ボタン制御更新
- **異常系**: エラーメッセージ、ステータス不変、ロールバック
- **境界値**: 最初のステータス、最後のステータス、権限境界

## [パターンD] ファイルアップロード

### 適用対象
- プロフィール画像アップロード
- ドキュメント添付
- CSVインポート
- 画像ギャラリー

### 必須確認観点

#### 1. ファイル形式の検証
- **許可された形式**
  - 指定された拡張子のファイルがアップロード可能
  - プレビュー表示（画像の場合）
- **許可されない形式**
  - 不正な拡張子はエラー
  - エラーメッセージの内容確認

#### 2. ファイルサイズの検証
- **サイズ制限**
  - 最大サイズ以下: 正常にアップロード
  - 最大サイズ超過: エラーメッセージ
  - エラー時、ファイルが破棄される

#### 3. プログレス表示
- **アップロード中**
  - プログレスバーまたはパーセント表示
  - キャンセルボタンの有効性
- **完了時**
  - 完了メッセージ表示
  - アップロードされたファイルの確認リンク

#### 4. 複数ファイル対応（該当する場合）
- **同時アップロード**
  - 複数ファイルを選択してアップロード
  - それぞれのプログレス表示
- **個別エラー**
  - 一部ファイルがエラーでも、他は正常にアップロード

#### 5. 期待結果の標準パターン
- **正常系**: ファイルアップロード成功、プレビュー表示
- **異常系**: エラーメッセージ、ファイル未保存、リトライ可能
- **境界値**: 最小サイズ、最大サイズ、0バイトファイル

## [パターンE] モーダル・ダイアログ

### 適用対象
- 削除確認ダイアログ
- 詳細情報モーダル
- 編集フォームモーダル
- 通知・アラート

### 必須確認観点

#### 1. モーダルの表示・非表示
- **表示**
  - トリガーボタンをクリック → モーダル表示
  - 背景がオーバーレイされる
  - フォーカスがモーダル内に移動
- **非表示**
  - 閉じるボタン（×）をクリック → モーダル非表示
  - 背景クリック（該当する場合） → モーダル非表示
  - ESCキー → モーダル非表示

#### 2. フォーカス制御
- **モーダル内のタブ移動**
  - Tab キーでモーダル内の要素のみを移動
  - モーダル外の要素にフォーカスが移動しない
- **モーダルを閉じた後**
  - 元のトリガー要素にフォーカスが戻る

#### 3. データの保持・破棄
- **編集モーダル**
  - 保存ボタン: データ保存、モーダル閉じる
  - キャンセルボタン: データ破棄、モーダル閉じる
  - 背景クリック: データ破棄（確認ダイアログ表示の場合あり）

#### 4. スクロール制御
- **モーダル表示中**
  - 背景のスクロール無効化
  - モーダル内のスクロールは有効
- **モーダルを閉じた後**
  - 背景のスクロール再開
  - スクロール位置が保持される

#### 5. 期待結果の標準パターン
- **正常系**: モーダル表示/非表示、データ保存/破棄の動作
- **異常系**: モーダル内のバリデーションエラー、モーダルは閉じない
- **境界値**: 長いコンテンツのスクロール、連続表示

すべてのパターンで共通して確認すべき異常系観点：

### 1. ネットワークエラー
- **オフライン状態**
  - 通信エラーのメッセージ表示
  - リトライボタンの提供
- **タイムアウト**
  - 適切なタイムアウト時間設定
  - タイムアウト時のエラーハンドリング

- **セッション切れ**

  - 404エラー画面
  - 一覧画面へ戻るリンク提供
- **削除済みデータの操作**
  - エラーメッセージ表示
  - 最新状態への再読み込み提示

### 5. ブラウザ互換性
- **対応ブラウザ**
  - Chrome, Firefox, Safari, Edge での動作確認
  - 古いブラウザでの適切なエラーメッセージ

## [共通：技術実装・環境規約]

このセクションは、エージェント実行時に参照される技術的な標準規約を定義します。

### 1. セレクタ戦略と優先順位

Playwright公式推奨に基づき、以下の優先順位でセレクタを選定してください。

#### 優先順位

1. **Roleベース** (`getByRole`) - 最優先
   - アクセシビリティ準拠
   - ユーザー視点での要素特定
   - 例: `getByRole('button', { name: '登録' })`
   - 適用対象: button, link, textbox, heading等

2. **Labelベース** (`getByLabel`) - フォーム要素に最適
   - 可読性が高い
   - 仕様変更に強い（UIテキストベース）
   - 例: `getByLabel('お名前')`
   - 適用対象: input, select, textarea

3. **TestIdベース** (`getByTestId`) - 一意性確保用
   - 開発チームがdata-testid属性を付与している場合に使用
   - 一意性が保証される
   - 例: `getByTestId('user-name-input')`
   - 適用対象: 汎用的に使用可能

4. **CSS/XPathセレクタ** - 最終手段
   - 脆弱性あり（DOMの変更に弱い）
   - 上記3つが使用できない場合のみ使用
   - 例: `#userName`, `.form-input[name="email"]`

#### セレクタ選定の判断フロー

```
要素にrole属性がある？
  → YES: getByRole を使用
  → NO ↓

フォーム要素でラベルがある？
  → YES: getByLabel を使用
  → NO ↓

data-testid属性がある？
  → YES: getByTestId を使用
  → NO ↓

CSSセレクタを使用（最終手段）
```

#### element-map.json の記載形式

セレクタ情報は以下の形式で記録してください：

```json
"selectors": {
  "roleBase": "getByRole('button', { name: '登録する' })",
  "labelBase": "getByLabel('お名前')",
  "testId": "getByTestId('submit-button')",
  "cssSelector": "button[type='submit']"
}
```

優先度の高い順に記載し、コード生成時は上から順に試行します。

### 2. プロキシ環境対応

社内QA環境へのアクセスにはプロキシ設定が必要です。

#### 設定方法

**PAC（Proxy Auto-Config）ファイルを使用する場合**:

```bash
# 環境変数に設定（pac+ プレフィックスが必須）
export HTTP_PROXY="pac+http://mrqa1:8888/proxy.pac"
export HTTPS_PROXY="pac+http://mrqa1:8888/proxy.pac"
```

**プロキシサーバーを直接指定する場合**:

```bash
export HTTP_PROXY="http://proxy.example.com:8080"
export HTTPS_PROXY="http://proxy.example.com:8080"
```

#### 重要事項

- **PACファイル使用時の注意**: `pac+` プレフィックスを必ず付与すること
- **認証が必要な場合**: `http://user:pass@proxy:8080` 形式で指定
- **設定タイミング**: Playwright MCP呼び出し前（Phase 2開始時）に環境変数を設定
- **適用範囲**: 設定後、全てのHTTP/HTTPS通信がプロキシ経由になる

#### 社内プロキシ（mrqa1）利用時の推奨手順

1. Phase 2開始前にプロキシ設定を環境変数にセット
2. 画面遷移前に接続確認（オプション）
3. ログイン処理が必要な場合は、プロキシ経由でアクセス可能か確認

### 3. 出力資産の構造化ルール

生成される画面分析素材は、以下の命名規則に従って整理してください。

#### 単一画面の場合

```
screen-analysis/
├── html-sources/
│   └── initial-page.html
├── screenshots/
│   ├── viewport.png          # 基本（必須）
│   └── full-page.png          # オプション
├── accessibility/
│   └── snapshot.txt
├── element-map.json
└── analysis-report.md
```

#### 複数画面遷移の場合

```
screen-analysis/
├── step-1-{page-name}/       # 例: step-1-list
│   ├── html-sources/
│   │   └── page.html
│   ├── screenshots/
│   │   ├── viewport.png
│   │   └── {action}.png      # 例: click-detail-button.png
│   └── element-map.json
├── step-2-{page-name}/       # 例: step-2-detail
│   ├── html-sources/
│   │   └── page.html
│   ├── screenshots/
│   │   └── viewport.png
│   └── element-map.json
└── flow-summary.md           # 全体の遷移フロー
```

#### 命名規則

- **ディレクトリ名**: `step-{N}-{page-name}` 形式（N は 1 から連番）
- **スクリーンショット基本名**: `viewport.png`（初期表示範囲）
- **フルページ**: `full-page.png`（必要時のみ）
- **操作後のスクショ**: `{action}.png`（例: `after-submit.png`, `error-state.png`）

#### 推奨する保存タイミング

- **初期表示**: 画面遷移直後、安定待ち完了後
- **操作前**: 特定の操作（ボタンクリック等）の直前
- **操作後**: 操作実行後、レスポンス完了後
- **エラー状態**: バリデーションエラー等の表示時

### 4. HTMLソースの取得と保存

#### 取得時のオプション設定

HTMLソースは以下のオプションで取得してください：

```typescript
playwright_get_visible_html({
  cleanHtml: true,        // 不要タグを削除
  removeScripts: true,    // スクリプトタグを除去
  removeComments: true,   // コメントを除去
  maxLength: 20000        // 最大文字数制限
})
```

#### 保存形式

```html
<!DOCTYPE html>
<!-- 取得日時: 2026-02-19 10:30:00 -->
<!-- URL: https://qa1.example.com/register -->
<!-- 取得時の設定: cleanHtml=true, removeScripts=true -->
<html>
...（整形されたHTML）
</html>
```

#### 部分HTML取得の推奨箇所

フォーム、モーダル、エラーメッセージエリア等、重要な部分は個別ファイルとして保存：

```
html-sources/
├── initial-page.html       # ページ全体
├── form-element.html       # フォーム部分
├── modal-content.html      # モーダル内容
└── error-area.html         # エラー表示エリア
```

### 5. 画面遷移後の安定待ち

SPA（Single Page Application）やAjax通信を含む画面では、遷移後の安定待ちが必須です。

#### 基本パターン

```typescript
// 方法1: ネットワークアイドル待機（推奨）
playwright_navigate({
  url: target_url,
  waitUntil: 'networkidle'
})

// 方法2: 特定要素の表示待ち
playwright_wait_for({ text: "ページタイトル" })
playwright_wait_for({ selector: "main" })

// 方法3: 固定時間待機（非推奨、最終手段のみ）
// waitForTimeout は使用しない
```

#### 待機戦略の選択基準

- **静的サイト**: `load` イベントで十分
- **SPA/Ajax**: `networkidle` 必須
- **段階的ローディング**: 主要要素の表示待ち併用
- **非同期データ取得**: 特定のデータ表示を待機

#### 複数画面遷移時の注意

各ステップで遷移後に必ず安定待ちを実行してください：

```
step-1: 遷移 → 安定待ち → スクショ
step-2: 遷移 → 安定待ち → スクショ
step-3: 遷移 → 安定待ち → スクショ
```

### 6. アクセシビリティツリーの活用

`browser_snapshot` で取得されるアクセシビリティツリーは、セレクタ選定の重要な情報源です。

#### 活用方法

1. **Role情報の抽出**: `getByRole` のセレクタ生成に使用
2. **Label情報の抽出**: `getByLabel` のセレクタ生成に使用
3. **階層構造の把握**: 要素間の関係性を理解
